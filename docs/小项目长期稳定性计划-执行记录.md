## 0. 本文档的定位

本文件不是总结报告，而是每一轮真实动作的流水账。  
后续 AI 每做完一轮事情，都必须把实际结果追加到本文件末尾。

这份记录的作用只有三个：

1. 让下一个 AI 知道上一轮到底做了什么。
2. 让主计划里的勾选有依据。
3. 防止“明明没验证、没修完、没留痕，却假装已经推进了”。

## 0.1 与主计划的关系

本文件不是接手入口，而是接手时必须按需查看的最新现场记录。

后续 AI 使用本文件前，必须先按 `docs/小项目长期稳定性计划-主计划.md` 的 `0.1 AI 最小接手读取协议` 确认当前进度。  
进入任何阶段前，必须按主计划 `0.2 阶段信息足够性门禁` 读取本文件最新记录。

读取规则：

1. 优先读取文件末尾最新记录，不要默认全量阅读。
2. 如果最新记录引用了某个 bug 编号、阶段结论或历史轮次，再回读对应旧记录。
3. 如果最新记录和主计划勾选状态不一致，先停止推进，补查原因。
4. 如果本文件没有真实执行记录，就按 `8. 第一轮记录前的明确要求` 执行。

追加规则：

1. 每轮只追加，不覆盖历史。
2. 文档调整、验证、修复、阶段结论都必须留痕。
3. 记录中的“下一步继续”必须能让下一个 AI 直接知道接下来做什么。

## 1. 使用总规则

1. 本文件只记录真实动作，不写空话。
2. 每一轮统一追加到文件末尾，不覆盖历史。
3. 每一轮必须写清楚：
   - 本轮目标
   - 实际做了什么
   - 涉及哪些文件
   - 实际验证结果
   - 当前是否能继续下一步
   - 还剩什么风险
4. 如果本轮没有改代码，也必须明确写“本轮未改代码，只做验证 / 只做文档 / 只做排查”。
5. 如果本轮发现了真实 bug，必须同步写入 `真实bug与风险清单`。
6. 如果本轮完成了主计划某个子项，才能回到主计划里打勾。
7. 不允许只改主计划勾选，不写执行记录。
8. 不允许把复杂运维体系重新写回本计划。

## 2. 什么时候必须写记录

出现以下任一情况，就必须新增一轮记录：

1. 做完一轮基线验证。
2. 确认了一个真实 bug。
3. 修完一个 bug。
4. 改完一个关键文档口径。
5. 做完一个阶段性结论。
6. 决定暂停、后置、放弃某个方向。

## 3. 什么时候允许主计划打勾

只有同时满足以下条件，才允许在主计划中打勾：

1. 实际动作已经完成。
2. 已经写入本执行记录。
3. 有明确验证结果。
4. 如果是 bug 修复，已经说明解决了哪个真实问题。

以下情况不允许打勾：

1. 只是分析过。
2. 只是怀疑可能修好了。
3. 只是改了文档，没有真实动作支撑。
4. 只是做了部分动作，还没有完成该子项。

## 4. 当前阶段的下一步

当前已完成 `P0`、`P1`、`P2-1`、`P2-2`，下一个 AI 接手时默认先做：

1. 先按主计划 `0.1 AI 最小接手读取协议` 确认当前进度。
2. 再按主计划 `0.2 阶段信息足够性门禁` 确认本轮信息足够。
3. 按真实 bug / 风险清单选择“继续观察”或“最小修复”对象。
4. 每轮只跑可承受的定向回归，不跑全量回归。
5. 新增真实问题再更新 [真实bug与风险清单-2026-04-13.md](C:/Users/徐逸飞/Desktop/file-hub/docs/真实bug与风险清单-2026-04-13.md)。
6. 每轮结束必须追加执行记录并回写主计划勾选。

## 5. 记录时必须写清楚的字段

每一轮最少要写清楚下面这些信息：

1. 这一轮属于什么类型
   - 基线验证
   - bug 排查
   - bug 修复
   - 文档调整
   - 阶段结论

2. 这一轮的目标是什么

3. 这一轮实际完成了什么

4. 这一轮涉及了哪些文件
   - 如果没改代码，要写“未改代码”
   - 如果只改文档，要写具体文档路径

5. 这一轮怎么验证的

6. 验证结果是什么
   - 通过
   - 失败
   - 部分通过
   - 暂无法验证

7. 是否发现真实 bug
   - 如果发现，要写编号或标题
   - 如果没发现，要写“本轮未发现新的真实 bug”

8. 是否阻断下一步
   - 阻断 / 不阻断
   - 为什么

9. 当前残留风险是什么

10. 下一步继续做什么
