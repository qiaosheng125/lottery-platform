## 一句话结论

项目已经从早期的 `SQLite + 单 worker` 保守模型，逐步演进到当前推荐的生产方案：

- `PostgreSQL + Redis + Gunicorn workers=2`

核心目标始终没变：

- 不能重复分票
- 不能串设备
- 不能残留 `assigned`
- 不能让文件计数和金额漂移

## 关键里程碑

### 第一阶段：打基础

这一阶段主要完成了系统骨架收口和基础稳定性提升：

- 梳理 A / B 模式核心流程
- 明确设备、会话、票据、中奖记录等核心模型
- 建立业务日口径
- 建立并发安全的基本原则
- 加入回归测试和活体并发测试基础

这一阶段最重要的产出不是功能本身，而是把“分票安全是最高优先级”这件事固定下来。

### 第二阶段：大量前后端 Bug 修复

这一阶段修了很多会直接影响日常操作的阻断性问题，主要集中在：

- 前端模板脚本损坏、乱码、按钮无响应
- API 路径错误
- 登录页和设备 ID 透传问题
- A / B 模式处理链路中的状态错误
- 后台管理页的异常处理缺失
- 导出、上传、结果页等多个页面的交互失真

这一阶段的价值在于把“能打开页面”推进到“页面真的能稳定工作”。

### 第三阶段：业务规则统一

这一阶段核心是把多个地方的口径统一到同一套规则：

- 业务日统一为 `12:00 -> 次日 12:00`
- A / B 模式超时确认逻辑统一
- B 模式客户端权限与设备规则统一
- 后台筛选、导出、统计都统一到业务日口径
- 历史保留策略统一到“只保留最近 30 天”

这一阶段的价值在于减少“同一份数据在不同页面看起来不一致”的问题。

### 第四阶段：高风险核心链路修复

这一阶段修的都是最关键、最危险的问题：

#### 1. PostgreSQL 分票并发上限穿透

修复前：

- 同一用户多设备并发时，虽然不会分到同一张票
- 但可能穿透每日上限或 B 模式处理中上限

修复后：

- PostgreSQL 分票前增加用户级 advisory lock
- 同一用户的并发分票会在事务内串行化

#### 2. 中奖图回填串票风险

修复前：

- `/api/winning/record`
- `/admin/api/winning/record`

只校验票归属和权限，没有校验 `oss_key` 是否真的属于当前票。

修复后：

- 强制校验 `oss_key` 和 `ticket_id` 的绑定关系
- 已检查记录仍优先返回 `403`

#### 3. 同业务日同名 TXT 并发重复导入

修复前：

- 应用层有重名检查
- 但并发上传时仍可能双写入

修复后：

- PostgreSQL 路径增加 advisory lock
- SQLite 路径增加进程内 guard

这一阶段的价值在于把真正可能影响数据正确性的风险点补上。

## 生产部署演进

### 旧认知

过去的注意事项里有一条：

- Gunicorn 必须单 worker

这句话现在需要拆开理解：

- 对 `SQLite` 开发环境，这句话仍然成立
- 对 `PostgreSQL + Redis` 生产环境，这句话已经不成立

### 当前推荐

当前项目推荐的生产配置是：

- `PostgreSQL`
- `Redis`
- `Gunicorn workers=2`
- `DB_POOL_SIZE=5`
- `DB_MAX_OVERFLOW=5`

同时已经补齐：

- Ubuntu 一键部署脚本
- Linux 严格压测脚本
- Linux 阶梯压测脚本
- PostgreSQL 空库 bootstrap 修复

## 压测结论

在 Linux 实测环境下，结论已经比较明确：

- `40` 设备档通过
- `60` 设备档开始首先暴露登录/会话超时

这说明：

- 当前没有证据表明 `workers=2 + PostgreSQL + Redis` 会导致重复分票
- 当前真正首先出现的问题是登录高峰，而不是核心票务正确性崩掉

所以当前唯一最该注意的风险点是：

- 大量设备同一时刻集中重新登录

## 现在项目最重要的结论

如果只记 5 条，就记这 5 条：

1. 生产环境不要再用 SQLite，改用 `PostgreSQL + Redis`
2. 生产 Gunicorn 起步配置用 `workers=2`
3. 当前成功档位没有看到重复分票、串设备、残留 `assigned`
4. 当前最大风险不是错票，而是登录高峰超时
5. 云服务器上线前后，重点盯登录耗时、会话建立、分票耗时，而不是先担心核心分票逻辑错乱
