# 基线验证清单（2026-04-13）

> 基线版本：`2775552`  
> 目的：在任何正式修改代码之前，先确认当前稳定基线在真实环境下到底哪里稳定、哪里有问题  
> 使用方式：按顺序逐项执行，每完成一项就在主计划里打勾，并把结果写入执行记录

## 0. 与主计划的关系

本文件不是接手入口，只在主计划确认当前阶段是 `P0-1 核心基线验证` 后读取。

后续 AI 使用本清单前，必须先完成：

1. 按 `docs/小项目长期稳定性计划-主计划.md` 的 `0.1 AI 最小接手读取协议` 确认当前进度。
2. 按主计划 `0.2 阶段信息足够性门禁` 确认信息足够。
3. 读取主计划 `6.2 P0-1 核心基线验证`，确认本阶段目标、边界、禁止事项和完成标准。

本清单只负责展开 `P0-1` 的验证动作。  
如果本清单与主计划冲突，以主计划为准。

使用本清单时必须同步：

1. 验证结果写入 `docs/小项目长期稳定性计划-执行记录.md`。
2. 真实发现的问题写入 `docs/真实bug与风险清单-2026-04-13.md`。
3. 完成对应验证后，回到主计划更新 `P0-1` 勾选项。

## 一、使用原则

1. 本清单的目标不是“证明系统没问题”，而是找出系统当前真实状态。
2. 发现问题后先记录，不要一边验证一边随手扩展设计。
3. 每一项都要写结果：
   - 通过
   - 失败
   - 部分通过
   - 暂无法验证
4. 没验证过的链路，不能假设稳定。
5. 本清单优先于任何“长期优化构想”。

## 二、验证输出格式

每完成一项，执行记录里至少要写：

- 验证项名称
- 验证环境
- 实际操作步骤
- 实际结果
- 是否发现 bug
- 是否阻断下一步

推荐记录格式：

```md
- 验证项：
- 操作步骤：
- 结果：通过 / 失败 / 部分通过 / 暂无法验证
- 问题说明：
- 是否阻断后续：
```

## 三、验证顺序

必须严格按以下顺序执行，不允许跳过主链路直接做边缘验证。

1. 启动与基础环境
2. 登录与会话
3. 管理员上传 TXT
4. 文件拆票与入库
5. A 模式主链路
6. B 模式主链路
7. 用户端统计与导出
8. 赛果上传与重算
9. 中奖图上传与记录
10. 调度器与定时任务
11. 后台页面基础可用性

## 四、详细验证清单

## 4.1 启动与基础环境验证

### 目标

确认系统在当前环境下能正常启动，数据库初始化和基础目录创建不出错。

### 必查项

- [ ] `app.py` 能正常创建应用
- [ ] `.env` 配置能被读取
- [ ] 数据库连接能建立
- [ ] SQLite / PostgreSQL 当前实际使用的是哪一个
- [ ] 上传目录是否会自动创建
- [ ] `uploads/images`
- [ ] `uploads/txt`
- [ ] `uploads/archive/txt`
- [ ] 启动时 scheduler 是否按预期启动

### 重点观察

1. 启动时报错是否与数据库初始化有关
2. 启动时报错是否与 scheduler 提前读取配置有关
3. 启动后首页和登录页是否可访问

### 通过标准

- 系统能稳定启动
- 没有明显启动级报错
- 基础目录齐全

## 4.2 登录与会话验证

### 目标

确认认证链路可用，登录、登出、心跳不会明显失效。

### 必查项

- [ ] 管理员账号可正常登录
- [ ] 普通账号可正常登录
- [ ] 错误密码时返回正确错误
- [ ] 禁用账号时返回正确错误
- [ ] 登出后会话失效
- [ ] `/auth/heartbeat` 可正常刷新会话
- [ ] 设备 ID 绑定逻辑是否符合预期
- [ ] 非管理员设备数限制是否生效

### 重点观察

1. 会不会出现“登录成功但实际没有有效会话”
2. 会不会出现“心跳正常返回但会话实际没续期”
3. 多设备登录时是否容易触发异常

### 通过标准

- 登录/登出/心跳都能稳定工作
- 会话不过早失效
- 设备数限制逻辑行为符合预期

## 4.3 管理员上传 TXT 验证

### 目标

确认上传入口本身正常，文件名校验、文件保存、重复文件限制都符合设计。

### 必查项

- [ ] 能成功上传一个合法 TXT 文件
- [ ] 非法文件名能被拒绝
- [ ] 空文件能被拒绝
- [ ] 文件编码异常能被正确报错
- [ ] 同业务日重复同名文件是否被拒绝
- [ ] 上传后磁盘文件是否实际落地
- [ ] 上传后 `uploaded_files` 记录是否生成

### 重点观察

1. 上传失败后是否留下脏数据
2. 上传失败后磁盘是否留下垃圾文件
3. 重复文件防护是否真实有效

### 通过标准

- 上传成功时数据和文件都正确
- 上传失败时不污染数据

## 4.4 文件拆票与入库验证

### 目标

确认上传后的拆票逻辑正确，票据数、金额、彩种、期号、截止时间等不漂。

### 必查项

- [ ] 文件名声明张数与实际拆票张数一致
- [ ] 文件名声明金额与实际票据金额一致
- [ ] 每行内容能被正确解析
- [ ] 彩种与玩法匹配校验有效
- [ ] 倍数与文件名倍数一致性校验有效
- [ ] `uploaded_files` 计数字段是否正确
- [ ] 新票据默认状态是否正确（`pending` / `expired`）

### 重点观察

1. 金额汇总是否漂
2. 张数是否漂
3. 过期票是否进入错误状态

### 通过标准

- 文件和票据数据一致
- 无明显计数漂移

## 4.5 A 模式主链路验证

- [ ] `/api/mode-a/previous` 可返回上一张票

### 重点观察

1. 是否会重复分到同一张票
