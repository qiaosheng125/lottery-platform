## 一、记录原则

1. 只有真实验证发现的问题才能写入。
2. 只记录“会影响系统稳定性、正确性、使用体验”的问题。
3. 不记录抽象口号。
4. 每个问题必须能回答：
   - 影响什么
   - 是否高频
   - 是否阻断
   - 应该先修还是后修

## 二、问题分级

## P0

满足任一条件即可归入 P0：

1. 直接影响核心业务正确性
2. 容易造成数据错乱
3. 会导致系统主要功能不可用
4. 会导致大面积用户无法正常处理
5. 会导致上传、分票、赛果、中奖图、导出等主链路异常

## P1

满足任一条件即可归入 P1：

1. 不一定马上阻断，但会持续影响长期稳定
2. 问题高频出现
3. 会明显增加人工处理成本
4. 问题一旦积累，后面会演变成 P0

## P2

满足任一条件可归入 P2：

1. 低频边界
2. 文案或提示问题
3. 局部体验问题
4. 不影响核心正确性

### [真实-P1-02] `P0-2` 定向回归出现方言锁与前端模板口径失配

- 分级：P1
- 所属模块：`services/ticket_pool.py`、`templates/client/dashboard.html`（及对应回归用例）
- 发现方式：`P0-2` 执行期间，运行 `python -m pytest tests/test_bug_fixes.py -q --maxfail=1` 的连续失败定位（2026-04-13）
- 复现步骤：
  1. 运行 `python -m pytest tests/test_bug_fixes.py -q --maxfail=1`
  2. 观察到前端模板相关断言连续失败（Mode B 端点调用、Mode A reset 分支、预览失败提示）
  3. 检查 `ticket_pool` 相关回归时发现：`_is_postgres=True` 但非 PostgreSQL 方言场景会误执行 `pg_advisory_xact_lock`
- 实际结果：
  - 回归用例与当前模板实现存在口径偏移，导致多条断言失败
  - `ticket_pool` 仅凭 `_is_postgres` 判断，缺少 DB 方言保护
- 预期结果：
  - 回归用例断言应与当前实现保持一致
  - advisory lock 仅在 PostgreSQL 方言执行，避免非 PG 方言误调用
- 影响范围：
  - 污染 `P0-2` 回归可信度，增加误报噪音
  - 增加分票链路在非 PG 方言测试场景的兼容风险
- 是否阻断：不阻断（可在 `P0-2` 内当轮闭环）
- 临时绕过方案：仅执行受影响子集定向回归，避免全量噪音放大
- 建议修复顺序：`P0-2` 当前轮直接修复并验证
- 当前状态：已修复并已验证（2026-04-13）
- 修复动作：
  - `services/ticket_pool.py`：`_acquire_postgres_user_assignment_lock` 增加 `bind.dialect.name == "postgresql"` 保护
  - `tests/test_bug_fixes.py`：同步 `client_dashboard` 相关断言与当前模板实现
- 验证结果：
  - `python -m pytest -q tests/test_bug_fixes.py -k "client_dashboard"` -> `31 passed`
  - `python -m pytest -q`（11 条 `ticket_pool` 关联定向用例）-> `11 passed`
  - `python -m pytest -q`（16 条本轮改动关联用例）-> `16 passed`
- 结论：本条属于“实现与回归口径漂移 + 方言保护缺失”，当前已闭环
