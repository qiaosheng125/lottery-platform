## 2. 项目是什么

这是一个数据文件管理与分发平台，核心业务流程是：

1. 管理员上传 TXT 文件。
2. 系统解析文件名和每一行内容。
3. 系统把文件拆成票据。
4. 用户设备以 A 模式或 B 模式处理票据。
5. 系统记录完成状态、统计金额和张数。
6. 管理员上传赛果后，系统进行中奖计算。
7. 用户或管理员上传中奖图片，形成中奖记录。
8. 管理端进行查看、统计、导出和基础设置。

这个项目不是内容管理后台，也不是复杂运维平台。  
它的核心是：**上传、拆票、分发、处理、赛果、中奖图、导出** 这一条业务链路长期稳定。

## 2.1 项目当前已知技术结论

根据当前代码和 README，可以确认这些事实：

1. 开发环境可用 `SQLite`，但只建议 `workers=1`。
2. 生产推荐 `PostgreSQL + Redis + Gunicorn workers=2`。
3. 当前已知更容易先暴露的问题是登录 / 会话高峰，不是分票正确性本身。
4. 分票正确性的关键目标包括：
   - 同一张票不能重复分给多个设备
   - 不能串设备确认
   - 不能残留 `assigned`
   - 文件计数和金额不能漂移

这意味着：

- 后续计划不能优先围绕复杂后台治理展开。
- 后续计划必须优先围绕认证、上传、分票、赛果、中奖图这些真实业务链路展开。

## 2.2 必须理解的业务术语

后续 AI 至少要正确理解这些词，不然计划和修复都会跑偏：

1. `UploadedFile`
   - 管理员上传的原始 TXT 文件记录。
   - 它有声明张数、声明金额、解析后的统计字段。

2. `LotteryTicket`
   - 从 TXT 每一行拆出来的票据记录。
   - 核心状态包括：`pending`、`assigned`、`completed`、`revoked`、`expired`。

3. `A 模式`
   - 设备按单张连续取票和处理。
   - 关键接口是：`next`、`current`、`stop`、`previous`。

4. `B 模式`
   - 设备按批次预览、下载、处理、确认。
   - 关键接口是：`preview`、`download`、`processing`、`confirm`。

5. `assigned 残留`
   - 票被分出去了，但没有被正常完成、释放、过期或回收，长期卡在 `assigned`。
   - 这是长期稳定性的重要风险信号。

6. `ResultFile / MatchResult`
   - 赛果文件和解析后的赛果记录。
   - 负责驱动中奖计算。

7. `WinningRecord`
   - 中奖图片和中奖记录的落地对象。
   - 必须和票据中奖状态保持一致。

8. `SystemSettings`
   - 管理后台的基础开关和关键参数来源。
   - 包括接单开关、模式开关、会话时长、日重置时间、公告等。

## 3. 本计划真正要达到什么

本计划只服务于当前项目的真实需要，不服务于“专业运维系统”式的复杂治理。

当前计划的目标只有四个：

1. 保证核心业务链路长期运行不容易出错。
2. 让管理员能快速看见异常。
3. 让定时任务可控、可判断。
4. 保持后台和文档简单，员工不需要学习复杂流程。

## 6.3 P0-2 核心业务链路修 bug

### 目标

只修真正影响长期运行的核心链路问题。

### 修复范围

优先关注：

1. 上传链路
2. 分票链路
3. A 模式主链路
4. B 模式主链路
5. 中奖图链路
6. 赛果导入和重算链路
7. 导出链路

### 默认修复顺序

在没有新的证据前，默认按这个顺序修：

1. 登录 / 会话
2. 上传 / 拆票
3. 分票 / 状态流转
4. 赛果 / 中奖图
5. 用户统计 / 导出
6. 调度器问题

### 开始前必须了解的信息

1. `P0-1` 已结束，且已经有真实 bug 条目。
2. 每个 bug 必须先有复现证据，再进入修复。
3. 修复时必须知道受影响的路由、service、model。
4. 不允许把“后面也许有用”的重构混入首批修复。

### 这一阶段的固定动作

每次只处理 1 到 2 个明确问题，并按以下顺序执行：

1. 从真实 bug 清单中选问题。
2. 说明为什么它是当前最高优先级。
3. 明确涉及文件。
4. 实施最小修复。
5. 做最小验证。
6. 更新执行记录。
7. 在主计划勾选或更新状态。

### 原则

1. 先修真实 bug，再谈长期治理。
2. 只修高频、主链路、可复现问题。
3. 不把“未来可能优雅”当成当前优先级。
4. 不允许一轮同时改太多主链路。

### 完成标准

核心链路的已知高频问题清零，至少达到：

- 主链路不会频繁报错
- 失败语义可判断
- 不会轻易造成数据错乱

### 对应勾选要求

- [x] 登录 / 会话首批问题清理完成（首批未发现已确认 bug，基线链路通过）
- [x] 上传 / 拆票首批问题清理完成（`真实-P1-01` 已闭环）
- [x] 分票 / 状态流转首批问题清理完成（`真实-P1-02` 已闭环）
- [x] 赛果 / 中奖图首批问题清理完成（首批未发现已确认 bug，基线链路通过）
- [x] 用户统计 / 导出首批问题清理完成（首批未发现已确认 bug，基线链路通过）
- [x] 调度器首批问题清理完成（首批未发现已确认 bug，基线链路通过）

### 6.3.1 P0-2 修复阶段总规则

#### 每轮修复前必须确认

1. 这个问题已经在 `真实 bug 与风险清单` 中登记。
2. 这个问题已经至少有一次真实复现或真实验证证据。
3. 这个问题属于当前主链路，而不是边缘体验问题。
4. 这一轮最多只处理 1 到 2 个明确问题。
5. 改动范围必须尽量集中，不允许一轮顺手重写多个模块。

#### 每轮修复的固定顺序

1. 先选 bug。
2. 再说明为什么它排在当前优先级。
3. 再确定涉及文件。
4. 再做最小修改。
5. 再做最小验证。
6. 再更新执行记录。
7. 最后才考虑在主计划里打勾。

#### 什么叫“这一轮修完了”

同时满足以下条件，才算这一轮修完：

1. 原问题已有明确修复动作。
2. 已经做过针对性验证。
3. 验证结果表明原问题已解决或主要现象已消失。
4. 已知残留风险已写入执行记录。

#### 什么叫“只是部分解决，不能打勾”

满足任一条，都不能直接打勾：

1. 只修了一个表面报错，但根因还在。
2. 只修了一个接口返回，数据库状态仍然会乱。
3. 只在一个入口正常，其他同类入口仍然异常。
4. 还没有做验证，只是从代码上“看起来应该好了”。

### 6.3.2 认证与会话问题修复映射

#### 为什么优先修

登录和会话是一切业务入口。入口不稳，后面上传、分票、导出、赛果的验证都会被污染。

#### 修复前必须先看

- `routes/auth.py`
- `services/session_service.py`
- `models/user.py`
- `app.py`

#### 优先关注的问题

1. 登录成功但会话不稳定。
2. 心跳成功但会话没续期。
3. 登出后会话仍然有效。
4. 设备数限制异常。
5. 强制下线、会话清理逻辑与登录状态不一致。

#### 什么算修完

1. 管理员与普通用户登录结果稳定。
2. 心跳和会话续期逻辑一致。
3. 登出后访问受保护资源会被正确拦截。
4. 设备限制逻辑符合预期，不误封、不失效。

#### 什么不算修完

1. 只是登录页能跳转，但会话还是会莫名丢。
2. 只是心跳接口返回 `success`，但实际数据库会话没更新。
3. 只是处理了某一种用户类型，另一种用户仍异常。

### 6.3.3 上传与拆票问题修复映射

#### 为什么优先修

上传与拆票是业务源头。源头错了，后面所有统计、分票、赛果结果都可能是错的。

#### 修复前必须先看

- `services/file_parser.py`
- `routes/admin.py`
- `models/file.py`
- `models/ticket.py`

#### 优先关注的问题

1. 上传成功但文件或票据没正确落库。
2. 文件名解析错误，导致期号、彩种、金额、倍数漂移。
3. 统计字段和实际票据不一致。
4. 上传失败后留下脏文件或脏记录。
5. 撤销文件后票据状态不同步。

#### 什么算修完

1. 合法文件上传后，文件记录和票据记录都正确落地。
2. 解析后的张数、金额、期号、彩种与样本一致。
3. 失败场景不会留下明显脏数据。
4. 撤销后文件与票据状态一致。

#### 什么不算修完

1. 只是前端提示成功，数据库仍不完整。
2. 只是让报错消失，但统计字段还是漂移。
3. 只是成功场景正常，失败场景残留问题还在。

### 6.3.4 分票与状态流转问题修复映射

#### 为什么优先修

分票和状态流转决定业务正确性，是这个系统最核心的内部机制。

#### 修复前必须先看

- `services/ticket_pool.py`
- `services/mode_a_service.py`
- `services/mode_b_service.py`
- `routes/mode_a.py`
- `routes/mode_b.py`
- `models/ticket.py`

#### 优先关注的问题

1. 重复分票。
2. 串设备确认。
3. `assigned` 长期残留。
4. `pending / assigned / completed / expired / revoked` 流转错误。
5. A / B 模式查询结果与数据库实际状态不一致。

#### 什么算修完

1. 同一张票不会被错误重复分配。
2. 不同设备不能错误确认彼此票据。
3. 主流程结束后不会留下大量异常 `assigned`。
4. A / B 模式展示结果与数据库状态一致。

#### 什么不算修完

1. 只是把某个接口报错改没了，但票据状态仍然会乱。
2. 只是单设备正常，多设备场景仍会串。
3. 只是 A 模式正常，B 模式同类问题还在。

### 6.3.5 赛果与中奖图问题修复映射

#### 为什么优先修

这条链路直接影响业务结果可信度。赛果或中奖图错了，管理员会直接怀疑系统结果。

#### 修复前必须先看

- `routes/admin.py`
- `routes/winning.py`
- `services/winning_calc_service.py`
- `services/oss_service.py`
- `models/result.py`
- `models/winning.py`
- `models/ticket.py`

#### 优先关注的问题

1. 赛果上传后不触发或不完成计算。
2. 重算行为错误。
3. 中奖图片错绑。
4. `WinningRecord` 与票据中奖状态不同步。
5. 已检查记录仍可被替换。
6. 替换图片时误删旧图或留下错图。

#### 什么算修完

1. 赛果上传、列表、详情、重算链路可正常工作。
2. 中奖计算状态可判断。
3. 非中奖票不能上传中奖图。
4. 图片、中奖记录、票据字段三者同步。
5. 已检查记录保护生效。

#### 什么不算修完

1. 只是图片上传成功，但记录和票据没有同步。
2. 只是赛果列表能看，重算还是不工作。
3. 只是处理单一上传路径，本地 / OSS 另一条链路仍异常。

### 6.3.6 用户统计与导出问题修复映射

#### 为什么优先修

它直接影响管理员对账和用户日常使用。虽然不是最底层，但会直接影响信任感。

#### 修复前必须先看

3. 无数据时误报成功。
4. 改密码成功但实际不可用。

#### 什么算修完

1. 统计结果与样本数据基本一致。
2. 导出文件结构和内容合理。
3. 无数据时返回明确提示。
4. 改密码后新旧密码行为符合预期。

#### 什么不算修完

1. 只是导出接口不报错，但文件内容还是错的。
2. 只是统计接口能返回，但数字明显不对。
3. 只是密码接口返回成功，但重新登录失败。

### 6.3.7 调度器与定时任务问题修复映射

#### 为什么排在后面

调度器对长期运行重要，但前提是上传、分票、认证这些主链路已经基本可用。

#### 什么算修完

## 6.4 P0-3 事务边界与失败语义最小收口

### 目标

让关键写接口在“主流程成功但副作用失败”时，不会表现混乱。

### 这里到底要做什么

只在关键接口上保留最小必要规则：

1. 主事务失败必须明确失败。
2. 副作用失败不能伪装成主事务失败。
3. 关键接口尽量统一返回可机读结果。
4. 同类错误的返回口径尽量一致。

### 重点接口

后续 AI 如果执行这一阶段，默认优先检查：

1. `routes/auth.py`
2. `routes/mode_a.py`
3. `routes/mode_b.py`
4. `routes/winning.py`
5. `routes/user.py`
6. `routes/admin.py` 中上传、赛果、设置相关入口

### 不要做什么

1. 不要把这里扩展成新的状态机项目。
2. 不要为了统一而强行重写大量老接口。
3. 不要引入大规模抽象层，只做最小收口。

### 完成标准

关键入口不再出现：

- 主流程成功却返回模糊失败
- 主流程失败却表面成功
- 同类错误不同入口口径乱飞

### 对应勾选要求

- [x] 登录 / 登出 / 心跳失败语义已检查
- [x] A 模式关键入口失败语义已检查
- [x] B 模式关键入口失败语义已检查
- [x] 中奖图关键入口失败语义已检查
- [x] 赛果关键入口失败语义已检查
- [x] 用户导出与改密码关键入口失败语义已检查

### 6.4.1 P0-3 的边界到底是什么

这一阶段不是“重构接口体系”，而是“把最容易让人误判的关键入口收口一下”。

只允许做下面三种事：

1. 把明显混乱的成功 / 失败返回改清楚。
2. 把主流程和副作用的成败区分清楚。
3. 把同类入口的错误口径收得更一致。

不允许做下面这些事：

1. 不允许新建一套通用响应框架。
2. 不允许为了统一，把整个项目所有接口都重写。
3. 不允许引入新的复杂状态枚举体系。
4. 不允许新增大量“规则文档驱动代码”设计。

### 6.4.2 这一阶段开始前必须满足的条件

后续 AI 只有在下面条件成立时，才能进入 `P0-3`：

1. `P0-1` 已经结束。
2. `P0-2` 至少完成了第一批真实 P0 问题修复。
3. 当前需要收口的是“关键入口返回口径混乱”，而不是“核心 bug 还没修”。

如果核心 bug 还在大量存在，就不该先做 `P0-3`。

  - `/admin/api/match-results/<result_id>/recalc`
  - `/admin/api/settings`

### 6.4.4 这一步具体看什么

每个关键入口都至少要看下面四件事：

1. 参数错误时，是否明确失败。
2. 权限不足时，是否明确失败。
3. 主流程失败时，是否真的返回失败。
4. 主流程成功但附带动作失败时，是否把问题说清楚，而不是把整个主流程伪装成失败。

### 6.4.5 什么叫“主流程”和“副作用”

后续 AI 执行这一阶段时，必须先区分：

1. 主流程
   - 这个接口最核心的业务动作
   - 例如：登录成功、取票成功、下载批次成功、中奖记录写入成功、赛果记录入库成功

2. 副作用
   - 不决定主流程是否成立，但会影响附加体验或后续便利
   - 例如：通知、附加统计刷新、非关键同步字段、非阻断型展示数据

只有先分清这两者，才知道该不该把整个接口当作失败。

### 6.4.6 各类入口的最小收口要求

#### 认证入口

最小要求：

1. 登录失败就明确失败。
2. 登录成功就必须保证会话真正建立。
3. 心跳不能“表面成功，实际没续期”。
4. 登出后不能“表面退出，实际还能访问”。

#### A / B 模式入口

最小要求：

1. 参数不合法必须直接失败。
2. 取票 / 下载失败不能伪装成成功但给空结果。
3. 查询类接口如果无数据，要和“出错”区分开。
4. 确认类接口必须让人看得出是“没数据、参数错、权限错、状态错”里的哪一种。

#### 中奖图入口

最小要求：

1. 非中奖票必须明确拒绝。
2. 权限不足必须明确拒绝。
3. 记录已检查的保护必须明确拒绝。
4. 图片上传成功但记录没同步，不能伪装成完整成功。

#### 管理端上传 / 赛果入口

最小要求：

1. 上传全部失败时必须明确失败。
2. 部分成功、部分失败时必须可判断，不允许混成一句模糊成功。
3. 重算触发失败必须明确暴露。
4. 设置更新失败时必须明确暴露。

### 6.4.7 什么算“口径一致”

这里说的一致，不是所有接口返回一模一样，而是满足这些最低要求：

1. 成功时能看出是成功。
2. 参数错误时能看出是参数错误。
3. 权限问题时能看出是权限问题。
4. 无数据时能看出是“没有数据”，不是“系统异常”。
5. 主流程失败时不能返回看起来像成功的数据结构。

### 6.4.8 什么算收口完成

只有同时满足下面条件，才算 `P0-3` 某一类入口收口完成：

1. 已明确该类入口当前存在的返回混乱点。
2. 已做最小修正。
3. 已对成功、参数错、权限错、无数据 / 状态异常这几类典型情况做过验证。
4. 执行记录里已经写清楚修了什么、验证了什么、还有什么没统一。

### 6.4.9 什么不算收口完成

满足任一条，都不能打勾：

1. 只是统一了文案，但成功 / 失败语义仍然乱。
2. 只是改了一个接口，其他同类接口还完全不同。
3. 只是把错误信息改短了，但没有解决“真假成功混淆”。
4. 为了统一，引入了大规模重构，反而扩大了风险。

### 6.4.10 P0-3 阶段结束时必须给出的结论

`P0-3` 结束时，后续 AI 必须明确写出：

1. 哪几类关键入口已经完成最小收口。
2. 哪些入口仍然存在口径差异，但目前不阻断进入 `P1`。
3. 是否已经足够支撑后续做“系统健康摘要”和“调度器最小可见性”。

如果这些结论写不出来，就说明 `P0-3` 还没真正完成。

## 7.2 P1-1 系统健康摘要（最小版）

### 目标

为管理员提供一个真正能看懂、真正有用的简洁健康摘要。

### 前提

必须在 `P0` 结束后再做。  
原因：没有真实 bug 和真实异常模式，健康摘要字段就会变成空想。

### 开始前必须了解的信息

执行这一阶段前，后续 AI 至少要知道：

1. 后台首页入口在 `routes/admin.py` 的 `dashboard` 和 `dashboard_data`。
2. 后台首页模板在 `templates/admin/dashboard.html`。
3. 当前后台首页已经展示在线用户、票池、速度、预计完成时间等业务统计。
4. 票据核心状态来自 `models/ticket.py` 的 `LotteryTicket.status`。
5. 文件聚合计数来自 `models/file.py` 的 `UploadedFile.pending_count`、`assigned_count`、`completed_count`。
6. 赛果计算状态来自 `models/result.py` 的 `MatchResult.calc_status` 和 `ResultFile.status`。
7. 中奖图记录来自 `models/winning.py` 的 `WinningRecord`。
8. 调度器状态只能从 `tasks/scheduler.py` 当前可获得的信息里取，不能为了健康摘要先扩展复杂调度器体系。

### 这一阶段的边界

`P1-1` 只做“摘要”，不做“治理平台”。

允许做：

1. 在现有后台首页增加一个简短健康摘要区域。
2. 在现有 `/admin/api/dashboard-data` 返回中增加少量聚合字段，或新增一个简单只读接口。
3. 用固定、少量、可解释的规则计算 `normal`、`warning`、`critical`。
4. 给管理员一句明确动作建议。

不允许做：

1. 不允许新增复杂巡检系统。
2. 不允许新增多层台账、确认、复核、派单流程。
3. 不允许把每条异常明细都铺到首页。
4. 不允许新增大量阈值配置入口。
5. 不允许把健康摘要和权限、发布、运维门禁绑定。

### 只允许包含的内容

建议只保留以下几类聚合异常：

1. 票据池异常：
   - 已过截止时间但仍为 `pending`。
   - 长时间 `assigned` 且没有完成。
   - `UploadedFile` 计数字段和实际票据状态明显不一致。

2. 赛果异常：
   - `ResultFile.status == 'error'`。
   - `MatchResult.calc_status == 'error'`。
   - `MatchResult.calc_status == 'processing'` 且持续时间超过 `P0` 观察到的正常范围。

3. 中奖图异常：
   - 票据标记中奖但缺少中奖记录。
   - 中奖记录存在但图片地址为空。
   - 已检查记录的保护逻辑在 `P0` 或 `P0-3` 中被标记为风险。

4. 调度器异常：
   - scheduler 未启动。
   - scheduler 已启动但注册任务数明显不对。
   - 关键任务最近验证失败，且这个失败已经在执行记录里出现过。

5. 会话异常：
   - 只在 `P0` 已确认登录 / 会话仍有残留风险时展示。
   - 默认不把会话统计做成独立运维页面。

### 状态分级口径

健康摘要最多分三档：

1. `正常`
   - 没有已知阻断问题。
   - 管理员可以继续正常上传、分票、赛果和中奖图操作。

2. `有提醒`
   - 存在少量异常或需要关注的数据。
   - 不阻断管理员继续操作，但需要留意某类业务。

3. `有严重问题`
   - 存在可能影响上传、分票、赛果或中奖图正确性的异常。
   - 管理员应暂停对应操作，并找技术处理。

不要再增加更多颜色、等级、权重、评分。

### 最小字段建议

健康摘要返回结构可以很小，默认只需要：

1. `status`
   - `normal` / `warning` / `critical`

2. `summary`
   - 一句话，例如“系统正常”或“发现 2 类异常，请先暂停赛果重算”

3. `items`
   - 最多 3 到 5 条摘要项。
   - 每条只包含：
     - `type`
     - `level`
     - `count`
     - `message`
     - `action`

4. `generated_at`
   - 当前北京时间。

不要在 `items` 里返回大量明细 ID。  
如确实需要定位，只返回对应管理页入口或筛选提示。

### 管理员视角要求

管理员只需要知道：

1. 系统正常 / 有提醒 / 有严重问题。
2. 哪一类问题有异常。
3. 大概有多少。
4. 现在该继续操作、暂停某类操作，还是找技术。

管理员不需要知道：

1. SQL 细节。
2. APScheduler 内部对象。
3. 异常栈。
4. 每条异常记录的完整字段。
5. 复杂阈值解释。

### 必须按顺序做的动作

1. 先回看 `P0-1`、`P0-2`、`P0-3` 的执行记录，确认真实异常类型。
2. 列出健康摘要只需要覆盖的 3 到 5 类异常。
3. 对每类异常写出数据来源、计算方式、展示文案和管理员动作建议。
4. 先在后端形成一个最小聚合结果。
5. 再把结果接到后台首页。
6. 做正常、提醒、严重三类最小验证。
7. 更新执行记录和本主计划勾选项。

### 实现位置建议

默认优先考虑下面位置：

1. `routes/admin.py`
   - 可以在 `dashboard_data` 中加入 `health_summary`。
   - 如果 `dashboard_data` 已经过重，可以新增 `/admin/api/health-summary` 只读接口。

2. `templates/admin/dashboard.html`
   - 只增加简短展示。
   - 不新增复杂筛选区。

3. `models/ticket.py`
   - 只作为查询对象。
   - 不为了摘要修改票据状态模型。

4. `models/result.py`
   - 只读取 `ResultFile.status`、`MatchResult.calc_status`、`calc_started_at`、`calc_finished_at`。

5. `models/winning.py`
   - 只读取中奖记录缺失或图片缺失这类明显异常。

### 数据查询规则

健康摘要查询必须遵守：

1. 聚合优先，不全表拉明细。
2. 优先按 `status`、`deadline_time`、`assigned_at`、`calc_status` 这些已有字段过滤。
3. 不要为了首页摘要引入昂贵跨表大查询。
4. 如果某类异常查询明显重，先不放进首页，记录为后续观察项。
5. 查询失败时，摘要本身可以显示“健康摘要暂不可用”，不能影响后台首页核心统计加载。

### 文案口径

文案必须短，且直接给动作建议：

1. 正常：
   - “系统正常，可以继续处理。”

2. 提醒：
   - “发现少量卡住票据，请留意处理速度。”

3. 严重：
   - “发现赛果计算异常，请暂停重复上传赛果并联系技术。”

不要写：

1. “系统健康评分 87 分”。
2. “请进入巡检台完成复核流程”。
3. “建议按 SOP 升级值班”。
4. “触发治理策略”。

### 验证要求

至少验证下面情况：

1. 没有异常时，后台首页显示正常。
2. 人为构造 `pending` 过期或 `assigned` 残留时，摘要出现提醒或严重问题。
3. 人为构造 `ResultFile.status == 'error'` 或 `MatchResult.calc_status == 'error'` 时，摘要能说清楚赛果异常。
4. 摘要查询异常时，不影响后台首页其他数据展示。
5. 未登录用户不能访问相关接口。

### 什么算完成

只有同时满足下面条件，才算 `P1-1` 完成：

1. 健康摘要字段清单已经确定。
2. 每个字段都有明确数据来源。
3. 每个异常项都有管理员动作建议。
4. 后台首页能看到简短摘要。
5. 正常、提醒、严重三类情况都验证过。
6. 执行记录写清楚哪些异常被纳入，哪些暂不纳入。

### 什么不算完成

满足任一条，都不能打勾：

1. 只画了页面，但没有真实数据来源。
2. 只返回一堆计数，但管理员看不懂该做什么。
3. 把健康摘要扩成复杂运维台。
4. 加入的查询明显拖慢后台首页。
5. 没有验证异常场景。

- [x] 健康摘要数据来源已确定
- [x] 健康摘要显示口径已确定
- [x] 管理员处理建议口径已确定
- [x] 后台首页最小展示已完成
- [x] 正常 / 提醒 / 严重三类验证已完成

### 7.2.1 P1-1 阶段结束时必须给出的结论

`P1-1` 结束时，后续 AI 必须明确写出：

1. 当前健康摘要包含哪几类异常。
2. 每类异常的数据来源是什么。
3. 哪些异常暂时没有纳入，原因是什么。

### 最小实现规则

1. 状态读取必须只读。
2. 状态读取失败时，接口应返回可机读失败摘要，不能让整个设置页崩掉。
3. 缺失任务只能提示，不自动补任务。
4. 不要在请求接口时重新启动 scheduler。
5. 不要在页面上暴露 Python 模块路径、异常栈、内部对象 repr。

### 验证要求

至少验证下面情况：

1. scheduler 正常启动时，状态为 `normal`。
2. scheduler 为 `None` 时，状态为 `critical`。
3. scheduler 存在但缺少预期任务时，能列出缺失任务。
4. 未登录用户不能访问相关状态接口。
5. 设置页加载失败时有明确提示，不影响其他设置项展示。

### 什么算完成

只有同时满足下面条件，才算 `P1-2` 完成：

1. 调度器最小状态字段已经确定。
2. 预期任务 ID 列表已经明确。
3. 管理员能在设置页或后台看到调度器状态。
4. 状态异常时有清楚动作建议。
5. 正常、未启动、任务缺失场景都验证过。
6. 执行记录写清楚没有做复杂任务控制台。

### 什么不算完成

满足任一条，都不能打勾：

1. 页面只写“调度器状态”，但没有真实读取 scheduler。
2. 暴露了一堆技术字段，管理员仍然不知道该做什么。
3. 增加了手动执行、重试、删除任务等控制能力。
4. 请求状态接口会顺手启动或重建 scheduler。
5. 没有验证 scheduler 异常场景。

5. 后台大查询主要集中在 `routes/admin.py`：

   - `/admin/api/match-results`
6. 用户侧统计和导出主要在 `routes/user.py`。
7. 当前项目是几十人使用的小项目，不按大规模数据平台标准设计。

### 处理方式

1. 先观察热点接口。
2. 必要时做有针对性的缓存、分页、归档。
3. 不做超前复杂治理。

### 这一阶段的边界

允许做：

1. 记录关键表数据量。
2. 记录热点接口响应时间。
3. 找出明显缺少分页或索引的查询。
4. 对真实变慢的接口做小范围优化。
5. 利用已有归档任务处理明确历史数据增长问题。

不允许做：

1. 不允许引入复杂数据仓库。
2. 不允许把所有查询都重写。
3. 不允许提前做分库分表。
4. 不允许为了“未来可能增长”增加大量缓存失效逻辑。
5. 不允许改变核心业务数据语义。

### 必须优先观察的对象

#### 数据表

1. `lottery_tickets`
   - 总量
   - `pending` 数
   - `assigned` 数

   - `revoked` 数

2. `uploaded_files`
   - 总量
   - 活跃文件数
   - 历史文件数

3. `winning_records`
   - 总量
   - 未检查数量
   - 缺图片数量

4. `match_results` / `result_files`
   - 总量
   - `processing` / `error` 数量

5. `user_sessions`
   - 总量
   - 已过期但未清理数量

#### 接口

1. 后台首页数据：`/admin/api/dashboard-data`

7. 用户面板数据：`/api/user/dashboard-data`

### 观察记录格式

每次观察必须在执行记录中写清楚：

1. 观察日期。
2. 数据库类型。
3. 各关键表大概行数。
4. 被测接口。
5. 测试数据规模。
6. 响应时间。
7. 是否已经影响管理员或用户操作。

   - `暂不处理`
   - `补分页`
   - `补索引`
   - `调整查询`
   - `启用或修正归档`
   - `继续观察`

### 优化顺序

如果观察发现真实变慢，默认按下面顺序处理：

1. 先确认是否缺少分页。
2. 再确认是否有明显 N+1 查询。
3. 再确认是否缺少现有字段索引。
4. 再确认是否可以收窄时间范围。
5. 再考虑使用已有归档任务。
6. 最后才考虑缓存。

缓存不是默认优先项。

1. 当前真实数据量。
2. 当前数据量的 3 到 5 倍模拟量。

4. 一个包含历史已完成票据的导出场景。

同时必须满足：

1. 已列出关键表数据量。
2. 已观察关键热点接口。
3. 已区分“真实慢”和“理论上可能慢”。
4. 已把需要处理的问题写入真实 bug / 风险清单。
