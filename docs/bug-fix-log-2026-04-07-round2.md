# 2026-04-07 持续排查修复记录（第二轮）

本轮继续针对全项目做确定性 bug 排查，重点补了归档清理、中奖链路、本地图片上传链路、结果重算链路，以及多个用户输入导致的 500 问题。

## 已修复问题

### 1. 归档清理误删仍需保留的数据

- `archive_old_uploaded_txt_files()` 之前会按文件上传时间直接删除整份 `UploadedFile` 和其关联票据。
- 现在改为：只有当该 `source_file_id` 下已经没有任何票据残留时，才允许删除原始 TXT 和文件记录。
- 这样不会把仍在 30 天保留期内的票据历史误删。

### 2. 辅助历史数据删除顺序存在外键风险

- `purge_old_auxiliary_records()` 之前先删 `ResultFile`，再删 `MatchResult`。
- 现在改为先删旧 `MatchResult`，再删已无引用的 `ResultFile`。
- 避免在外键约束更严格的数据库下直接失败。

### 3. 过期票中奖计算和展示不完整

- `expired` 票现在参与中奖计算，`revoked` 仍然不参与。
- 管理员中奖管理和中奖导出会显示 `expired` 中签票，并标记为“已过期未出票”。
- 用户自己的中奖记录仍然只显示 `completed` 中签票，不显示 `expired`。

### 4. 中奖重算后会残留脏数据

- 某张票从中奖变为未中奖时，之前旧的 `WinningRecord`、中奖图片、金额字段可能残留。
- 现在重算为未中奖或计算异常时，会同步清理：
  - `winning_gross`
  - `winning_amount`
  - `winning_tax`
  - `WinningRecord`
  - `ticket.winning_image_url`
  - 本地旧图片 / OSS 旧对象

### 5. 管理员上传中奖图后前后端状态不同步

- 管理员上传图片或回填 OSS 地址时，之前可能只更新了 `LotteryTicket`，没有保证同步创建/更新 `WinningRecord`。
- 现在管理员两条中奖图片接口都会创建或更新 `WinningRecord`，并回写 `record` 给前端。
- 前端当前行会立即更新 `winning_record_id`，无需刷新就能继续“标记已检查”。

### 6. 同 key 重传中奖图片会误删新图

- 在 OSS 使用固定 key 的情况下，旧逻辑会把刚上传的新对象当旧图删掉。
- 现在只有在“旧图目标”和“新图目标”不一致时才删除旧图。
- 用户端和管理员端都已修正。

### 7. 赛果上传/重算在调度器缺失时会假成功

- 之前如果调度器没启动，上传赛果或点击重算会返回成功，但后台不会真正执行计算。
- 现在改为：
  - 有调度器时继续异步执行
  - 没调度器时直接同步执行
- 同时在重新进入 `pending` 前清空旧的中奖统计字段，避免页面继续显示上一次结果。

### 8. 本地模式中奖图片上传链路不完整

- 本地模式预签名 URL 原先返回了错误的上传地址。
- 现在补齐了 `/api/winning/upload-local`，并和预签名接口对齐。
- 同时修复管理员和用户在本地模式下都能正常上传中奖图片。

### 9. 本地上传接口权限过宽

- 本地上传接口之前只校验登录，不校验该票是否属于当前用户，也不校验 `key` 是否与票匹配。
- 现在上传必须同时满足：
  - 带 `ticket_id`
  - 当前用户对该票有权限
  - `key` 与该票的预期 key 完全匹配

### 10. 管理员导出当日 CSV 会触发 `NameError`

- `/admin/api/tickets/export` 使用了 `timedelta`，但模块顶部未导入。
- 真实点击导出时会抛 `NameError`。
- 现已补齐导入，并验证业务日窗口筛选正常。

### 11. 多个接口遇到非法整数参数会直接 500

- 修复了以下入口的非法整数参数问题，统一改为返回 400：
  - `GET /api/mode-a/previous` 的 `offset`
  - `POST /api/mode-b/confirm` 的 `ticket_ids`
  - `GET /admin/api/files` 的 `page/per_page`
  - `GET /admin/api/files/<id>/detail` 的 `page/per_page`
  - `GET /admin/api/winning` 的 `page/per_page`
  - `POST /admin/api/users` 的 `max_devices`
  - `PUT /admin/api/users/<id>` 的 `max_devices`

### 12. 多个 JSON 接口在空 body 时会直接 500

- 修复了以下接口在 `Content-Type: application/json` 但 body 为空或解析失败时，因 `None.get(...)` 触发 500 的问题：
  - `POST /api/device/register`
  - `POST /api/user/change-password`
  - `POST /api/mode-b/confirm`
  - `POST /api/winning/record`
  - `PUT /admin/api/settings`
  - `POST /admin/api/winning/record`
- 现在统一会回到正常的业务错误响应，而不是直接抛异常。

### 13. 登录接口在空 JSON body 时响应不稳定

- `/auth/login` 之前在 JSON 请求体为空时，可能直接抛出解析错误。
- 现在会稳定返回登录失败 JSON，而不是框架级 400/HTML 响应。

### 14. 设备注册可抢占其他用户的设备 ID

- 之前任意已登录用户只要提交相同 `device_id`，就能把 `DeviceRegistry.user_id` 改成自己。
- 现在如果该设备 ID 已属于其他用户，会直接返回 409，阻止设备归属被劫持。

### 15. 管理员回填中奖图的“已检查”报错文案是乱码

- `/admin/api/winning/record` 在记录已检查时，返回的错误文案本身是乱码。
- 现已恢复为正常中文提示。

### 16. A 模式停止接单在并发下会假成功

- `stop_receiving()` 之前即使 `finalize_ticket()` 因并发失败，接口仍然返回“已停止接单，当前票已完成/已过期”。
- 现在会检查最终落库结果，失败时返回“当前票状态已变化，请刷新后重试”。

### 17. 重复期号赛果会更新到旧记录

- `parse_result_file()` 之前对同一期号使用 `first()`，如果库里历史上已经有多条 `MatchResult`，会更新到最旧那条。
- 现在改为按 `uploaded_at desc, id desc` 选择最新记录更新，符合“最新上传生效”的规则。

### 18. 赛果上传允许空文件名

- `/admin/match-results/upload` 之前只检查了有没有 `file` 字段，没有校验文件名是否为空。
- 现在空文件名会直接返回 400，不再创建脏的结果文件记录。

### 19. 系统设置接口缺少关键字段校验

- `/admin/api/settings` 之前几乎不校验类型和值范围。
- 现在补了以下校验：
  - `session_lifetime_hours` 必须是 `1-24` 的整数
  - `daily_reset_hour` 必须是 `0-23` 的整数
  - `mode_b_options` 必须是非空的正整数数组
- 同时对 `mode_b_options` 做去重和标准化，避免保存脏配置。

### 20. 用户中奖图片上传允许空文件名

- `/api/winning/upload-image/<ticket_id>` 之前未校验 `file.filename`，空文件名也会继续走图片处理逻辑。
- 现在空文件名会直接返回 400，与管理员上传接口保持一致。

### 21. 用户可绕过“已检查”限制直接换图

- 用户端 `/api/winning/record` 之前不会检查 `WinningRecord.is_checked`。
- 结果是管理员已经标记“已检查”的记录，用户仍可通过 OSS 回填接口直接替换图片，绕过 `/upload-image` 的限制。
- 现已统一拦截，返回 403。

### 22. 已检查记录仍可继续申请 presign，留下孤儿文件

- 用户端 `/api/winning/presign` 和管理员端 `/admin/api/winning/<ticket_id>/presign` 之前即使记录已检查，仍然会返回 presign URL。
- 这会导致图片先上传到 OSS 或本地，最后在 `/record` 被拒绝，留下孤儿对象/文件。
- 现在 presign 和本地上传入口都会提前拦截已检查记录。

### 23. 文件列表状态和筛选长期不准确

- `UploadedFile.status` 之前基本只在“撤回”时更新，正常完成或过期后数据库里仍常常保持 `active`。
- 结果是后台文件列表的状态筛选不准，撤回按钮也会对已完成/已过期文件继续显示。
- 现已改成统一按实时派生状态返回：
  - `revoked`
  - `exhausted`
  - `expired`
  - `active`
- 管理后台 `/admin/api/files` 的 `status` 筛选也已改为按派生状态生效。

### 24. 最大设备数限制可被旧设备会话绕过

- 登录时判断“同设备已有会话”原先不带活跃时间筛选。
- 这意味着只要拿一个历史很久以前的旧 `device_id`，也可能绕过当前的最大设备数限制。
- 现已统一只把活跃窗口内的同设备会话视为“existing”。

### 25. 新会话的过期时间不跟随数据库设置

- 登录限制和会话清理读的是数据库里的 `session_lifetime_hours`。
- 但 `create_session()` 之前仍然读配置常量 `SESSION_LIFETIME_HOURS`。
- 结果是设置页改了无活动超时时间后，新会话的 `expires_at` 实际不会跟着变。
- 现已统一优先读取数据库设置，失败时才回退到配置默认值。

### 26. 历史清理会删记录，但不会删中奖图片

- `archive_old_tickets()` 和 `archive_old_uploaded_txt_files()` 之前删除旧票据或 `WinningRecord` 时，不会同步删除中奖图片。
- 本地模式会留下孤儿图片文件，OSS 模式会留下孤儿对象。
- 现已在历史清理时同步删除对应图片。

### 27. `daily_reset_hour` 设置保存了，但业务日和调度都不会用

- 之前 `daily_reset_hour` 只是能保存到数据库，但：
  - 业务日 helper 仍然硬编码按 `12:00`
  - 每日会话重置任务也仍然硬编码在 `12:00`
- 结果是设置页改这个值，系统实际行为完全不变。
- 现已修复：
  - 业务日 helper 改为读取数据库里的重置小时
  - 调度器启动时按该小时注册每日重置任务
  - 设置页更新该值时会同步重排每日重置任务

### 28. 用户文件状态长期失真，后台筛选不准

- `UploadedFile.status` 之前基本只在撤回时更新，已完成或已过期文件常年仍是 `active`。
- 现在改为实时派生文件状态，并让后台文件列表的 `status` 筛选按派生状态生效。

### 29. 历史旧设备会话可绕过最大设备数限制

- 登录时判断“同设备已有会话”原先不区分是否仍在活跃窗口内。
- 历史很久以前的旧会话也会被当成当前设备，从而让最大设备数限制判断失真。
- 现已统一只认活跃窗口内的同设备会话。

### 30. 新会话过期时间不跟随数据库超时设置

- 新会话 `expires_at` 之前仍使用配置常量 `SESSION_LIFETIME_HOURS`。
- 设置页修改“无活动超时”后，登录限制和清理逻辑会变，但新建会话的过期时间不会跟着变。
- 现已统一优先读取数据库设置。

### 31. 活跃用户不会续期 `expires_at`，会被固定时长踢下线

- 系统原来虽然有 `last_seen` 和“无活动超时”配置，但 `expires_at` 只在登录时生成一次。
- 后续正常请求和 `/auth/heartbeat` 都不会续期 `expires_at`。
- 结果是活跃用户也会在固定小时后被强制登出，不符合“无活动超时”的语义。
- 现已修复：
  - 正常请求会按当前超时配置刷新 `expires_at`
  - `/auth/heartbeat` 会同步刷新 `expires_at`
  - `touch_session()` 也改为同时刷新 `last_seen` 和 `expires_at`

### 32. 用户中奖记录在自定义业务日小时下会重新变成错口径

- `routes/winning.py` 里的“我的中奖记录”之前仍然写死按 `12:00` 切业务日。
- 在默认配置下没问题，但一旦 `daily_reset_hour` 改成其他小时，这里会重新统计错误。
- 现已改为统一走 `get_business_window()`。

### 33. 中奖图片接口可被普通票伪造成“中奖票”

- 用户端和管理员端的中奖图片相关接口，之前只校验票据归属或管理员权限，没有校验票据是否真的被系统判定为中奖。
- 结果是只要知道票 ID，就有机会通过 presign、图片上传或 OSS 回填，把一张普通票错误地写成 `is_winning=True` 并生成 `WinningRecord`。
- 现已统一修复：
  - 用户端 `/api/winning/presign`
  - 用户端 `/api/winning/upload-local`
  - 用户端 `/api/winning/record`
  - 用户端 `/api/winning/upload-image/<ticket_id>`
  - 管理员端 `/admin/api/winning/<ticket_id>/presign`
  - 管理员端 `/admin/api/winning/record`
  - 管理员端 `/admin/api/winning/<ticket_id>/upload-image`
- 这些接口现在都会先验证该票 `is_winning=True`，未中奖票直接返回业务错误。

### 34. 非法 `ticket_id` 会在中奖图片接口打成 500

- 多条中奖图片接口之前直接对外部输入做 `int(ticket_id)`。
- 一旦前端或调用方传入 `abc`、空串之类的非法值，会直接抛 `ValueError`，导致接口 500。
- 现已统一改为显式校验并稳定返回 400，不再把无效参数打成服务器异常。

### 35. B 模式用户可直接调用 A 模式接单接口

- 之前只给 `/api/mode-b/*` 加了模式限制，`/api/mode-a/*` 仍然只校验登录状态和接单开关。
- 结果是 B 模式用户理论上可以直接调用 A 模式的 `next/current/stop/previous` 接口，绕过前端页面限制。
- 现已补齐 `mode_a_required`，A 模式接口和 B 模式接口一样都做服务端模式校验。

### 36. 管理员可写入非法 `client_mode` 和错误布尔开关值

- 创建/编辑用户时，`client_mode` 之前没有校验，任意字符串都能写入数据库。
- 这会让登录跳转、模式权限判断、桌面端限制等行为进入未定义状态。
- 同时，`is_active` / `can_receive` 之前直接用 `bool(value)`，如果前端传的是 `"false"`、`"0"` 这类字符串，反而会被当成 `True`。
- 现已修复：
  - `client_mode` 只能是 `mode_a` 或 `mode_b`
  - `is_active` / `can_receive` 只接受明确布尔值，或可解析的 `true/false/1/0`
  - 非法值统一返回 400，不再静默写错

### 37. 管理员删除已有业务历史的用户会触发数据完整性风险

- 后台“删除用户”之前是直接硬删。
- 但用户可能已被票据、上传文件、中奖记录、赛果记录、审计日志等历史数据引用。
- 在 SQLite 下这类问题不一定立刻暴露，换到更严格的数据库约束后更容易直接报错。
- 现已改为安全删除：
  - 只有完全没有业务历史引用的账号才允许删除
  - 一旦存在历史业务数据，接口返回 409，并提示改用“禁用账号”

### 38. B 模式处理中批次会把同一分钟内的不同下载合并

- B 模式恢复“处理中批次”列表时，之前按 `彩种 + 截止时间 + 分配时间(分钟级)` 分组。
- 如果同一设备在同一分钟里连续下载两次，而且彩种与截止时间又刚好相同，两批会被错误合并成一个批次。
- 这会导致前端显示张数错误，甚至确认完成时把本不属于同一批的票一起确认。
- 现已修复：
  - 分组键改为 `设备ID + 彩种 + 截止时间 + assigned_at 精确时间`
  - 同一分钟内的不同下载批次不会再被错误合并

### 39. B 模式短时间连续下载会让批次时间碰撞风险升高

- 你的使用场景里不需要上 `batch_id` 这种大改，但 B 模式仍需要一个轻量兜底，避免同设备短时间连续下载时碰到完全相同的批次时间。
- 现已做两层保护：
  - 后端：同设备同用户的 B 模式下载，如果本次 `assigned_at` 与当前处理中批次撞值，会自动顺延 `1 微秒`
  - 前端网页端：B 模式“下载”按钮增加 `1 秒` 冷却，防止普通用户连续点按钮
- 这样不用改协议，也能把你担心的批次碰撞风险压到很低。

### 40. 文件撤回/公告/票池关闭的实时推送没有真正同步到用户页状态

- 后端之前已经会推送 `file_revoked`、`announcement`、`pool_disabled` 这些实时事件。
- 但用户页本身没有监听这些自定义事件，也就不会刷新当前票、处理中批次、统计和公告状态。
- 结果是管理员撤回文件后，用户端可能还停留在旧界面，直到手动刷新才恢复一致。
- 现已修复：
  - `socket_client.js` 会把 `announcement` 和 `pool_disabled` 也分发成自定义事件
  - 用户页会监听 `announcement`、`pool_disabled`、`file_revoked`
  - 收到撤回事件时会刷新处理中批次、票池和统计
  - 如果当前 A 模式票正好属于被撤回文件，会立刻清掉当前票并退出接单状态

### 41. 多个 JSON API 在未登录时会返回 302/HTML，而不是 JSON 错误

- 一批面向前端 `fetch` 的接口之前装饰器顺序写反了：`@login_required` 在外层，`@login_required_json` 在内层。
- 结果是未登录或会话已失效时，接口先触发 Flask-Login 的重定向，返回的是 302/HTML，而不是前端可处理的 JSON 错误。
- 这类问题会让前端在拿 JSON 时直接失败，表现成莫名其妙的网络错误。
- 现已统一修复：
  - `/api/pool/status`
  - `/api/mode-a/*`
  - `/api/mode-b/*`
  - `/api/user/daily-stats`
  - `/api/user/change-password`
  - `/api/winning/*`
  - `/auth/heartbeat`
- 这些接口现在未登录时都会稳定返回 JSON `401`。

### 42. 管理员创建/重置用户密码缺少后端最小长度校验

- 用户自己改密码时后端原本就要求至少 6 位，但管理员创建用户和后台重置密码只靠前端提示，没有服务端校验。
- 这意味着直接调接口可以创建或写入弱密码。
- 现已统一补上后端校验：管理员创建用户和重置用户密码都要求密码至少 6 位。

### 43. 管理员禁用账号后，用户不会被立即踢下线

- 之前管理员把用户 `is_active` 改成 `false` 时，数据库状态会变，但用户当前会话不会立刻失效。
- 实际表现是：用户可能还停留在页面上，直到下一次请求才收到 `403`。
- 现已修复：
  - 管理员禁用用户时，会立即清掉该用户所有会话
  - 同时推送 `force_logout`，让在线客户端立刻退出
  - 并记录一条对应的审计日志

### 44. 系统设置里的布尔开关缺少后端校验

- 设置接口里的 `registration_enabled`、`pool_enabled`、`mode_a_enabled`、`mode_b_enabled`、`announcement_enabled` 之前没有做后端布尔解析。
- 如果前端或外部调用方传入的是 `"false"`、`"0"` 之类字符串，系统会把它们直接写进数据库，导致开关状态失真。
- 现已统一修复：
  - 这些字段只接受明确布尔值，或可解析的 `true/false/1/0`
  - 非法值会返回 `400`
  - 合法字符串值会先规范化再落库

### 45. 设备接口缺少 JSON 登录态语义和基础格式校验

- 网页端设备登记/改名走的是 `fetch('/api/device/*')`，但接口之前没有 `login_required_json`。
- 一旦会话失效，前端拿到的是 302/HTML，不是 JSON 错误。
- 同时，`device_id` 和 `device_name` 也几乎没有后端格式限制。
- 现已修复：
  - `/api/device/register` 和 `/api/device/<device_id>/name` 未登录时稳定返回 JSON `401`
  - `device_id` 只允许字母、数字、连字符、下划线，长度不超过 50
  - `device_name` 长度不超过 20

### 46. 文件上传结果缺少原始文件名，前端队列对位不稳

- 后台文件上传接口返回的 `results` 项之前没有 `filename` 字段。
- 当前前端虽然还能靠数组顺序兜底，但一旦后续顺序变化、并发处理或重构，上传队列就可能把成功/失败消息对错文件。
- 现已修复：`process_uploaded_file()` 无论成功还是失败都会回传原始 `filename`，让前端能稳定按文件名对位。

### 47. SQLite 下 A 模式单张分配在截止边界会过早返回“无票”

- SQLite 单张分配虽然已经在 `UPDATE ... WHERE status='pending' AND deadline_time > :now` 做了截止保护，但之前第一张候选票只要在 `SELECT` 和 `UPDATE` 之间失效，就会直接 `return None`。
- 实际影响是：哪怕后面还有别的可分票，A 模式这次请求也会被误判成“没票了”，属于核心分票链路的错误早停。
- 现已修复：SQLite 分支改成和 Postgres 一样的有限重试语义；当前一张票在 guarded update 阶段失败时，会回滚并继续尝试下一张候选票，而不是直接结束。

### 48. Postgres 下 B 模式下载漏掉了处理中上限控制

- `assign_tickets_batch()` 之前只在 SQLite 路径里检查 `max_processing`，Postgres 生产路径完全没做这层限制。
- 实际影响是：线上 B 模式用户在 Postgres 下可以超出“处理中票数上限”继续拿票，而且请求量被自动收缩时也拿不到任何调整提示。
- 现已修复：Postgres 路径补齐了与 SQLite 一致的 `max_processing` 检查和自动收缩提示；达到上限时会拒绝继续分票，接近上限时会自动把本次请求张数收缩到剩余额度，并返回调整消息。

### 49. B 模式下载服务层会把“没票/今日上限”误报成“处理中上限”

- `download_batch()` 之前只要用户配置了 `max_processing_b_mode`，一旦底层没拿到票，就直接返回“已达到处理中票数上限”。
- 这会把几种完全不同的情况混为一谈：票池真的空了、今日处理上限到了、以及处理中上限到了。
- 现已修复：服务层现在会优先透传 `assign_tickets_batch()` 返回的明确限制原因；如果没有明确原因，再分别检查今日上限和处理中上限，只有真的达到处理中上限时才返回对应错误，否则返回“当前票池无可用票”。

### 50. Postgres 批量标完成时文件计数可能按错票据集合更新

- `complete_tickets_batch()` 的 Postgres 分支之前先批量 `UPDATE lottery_tickets ... status='completed'`，但后面的 `uploaded_files` 计数更新仍然按原始请求的 `ticket_ids` 整批统计。
- 这意味着只要请求里混进了已完成、非当前用户或并发失败的票，文件计数就可能被多减/多加，和真实完成结果不一致。
- 现已修复：Postgres 分支改成只按 `UPDATE ... RETURNING id` 真正完成成功的票更新文件计数，并以这批真实完成的票数作为返回值。

### 51. Postgres 下“部分完成其余过期”确认链路也会按错票据集合更新计数

- `finalize_tickets_batch()` 的 Postgres 分支之前先用一次 `SELECT` 预筛可操作票，再分别执行 completed / expired 更新，但两个 `uploaded_files` 计数更新都还是按预筛后的 `valid_*_ids` 整批统计。
- 这意味着如果某些票在预筛后到真正 `UPDATE` 前已经被别的请求改走，文件计数仍会把这些没真正更新成功的票算进去。
- 现已修复：completed 和 expired 两段都改成只按各自 `UPDATE ... RETURNING id` 真正成功的票更新文件计数，并以这两批真实成功的票数作为返回结果。

### 52. Postgres 批量标完成链路缺少 `RETURNING id`

- 在把 `complete_tickets_batch()` 改成按真实成功票更新计数后，这条 SQL 仍然漏了 `RETURNING id`，代码却已经开始按 `fetchall()` 读取更新结果。
- 这在真实 Postgres 下会拿不到完成成功的票 ID，属于一处不完整修复。
- 现已补上：批量标完成 SQL 现在明确带 `RETURNING id`，和后续文件计数更新逻辑保持一致。

### 53. A 模式在 Redis 队列堆积脏票时可能误报“无票”

- Postgres 下 A 模式单张分配优先从 Redis `pool:pending` 取票，但之前如果 Redis 前面堆了很多已经失效的旧 `ticket_id`，代码会一直消耗这批脏 ID。
- 由于分配循环只有固定 `10` 次尝试，脏 ID 足够多时，即使数据库里还有真正可分的 `pending` 票，也可能提前返回“无票”。
- 现已修复：每轮尝试都会在消费 Redis 结果的同时补做一次数据库兜底选票；即使 Redis 弹出的是脏 ID，也不会阻断同轮从数据库拿到真正可分的票。

### 54. 过期任务之前不会把失效的 `pending` 票从 Redis 队列清掉

- `expire_overdue_tickets()` 之前只会改数据库状态和同步文件计数，不会把已经过期的 `pending` 票 ID 从 Redis `pool:pending` 里清掉。
- 这正是 A 模式 Redis 队列积脏的一个主要来源，会持续放大“前面全是旧票 ID”的问题。
- 现已修复：过期任务在提交数据库事务后，会把本轮变成 `expired` 的 `pending` 票从 Redis 队列中做 best-effort 清理。

### 55. B 模式确认链路对重复 `ticket_id` 不够稳健

- `finalize_tickets_batch()` 之前直接按传入列表顺序处理票据，没有先去重。
- 在 SQLite 路径下，如果请求里带了重复 `ticket_id`，同一张票可能被重复记完成、重复累加 `completed_count`，文件计数也会被带偏。
- 现已修复：批量确认入口先按原顺序去重 `ticket_ids`，后续 completed / expired 逻辑都只处理唯一票据。

### 56. A/B 模式分票入口之前缺少设备号格式校验

- `device_id` / `device_name` 会直接写入票据分配记录、处理中批次恢复和管理员统计分组，但 A 模式入口和 B 模式下载入口之前没有统一做后端格式校验。
- 这意味着异常或脏设备号可以直接进入核心分票数据面，后续会污染批次分组、设备统计和当前票查询。
- 现已修复：A 模式 `/api/mode-a/*` 和 B 模式 `/api/mode-b/download` 现在都统一校验 `device_id` 只能包含字母数字/`-_` 且长度不超过 64，`device_name` 长度不超过 128。

### 57. “同日同名文件拦截”之前能被大小写差异绕过

- 你定义的规则是“同一业务日内同名 txt 视为重复上传”，但之前后端按原始字符串直接比较文件名。
- 在 Windows 场景里，`CaseName.TXT` 和 `casename.txt` 实际上仍然是同名文件，这种只改大小写的重复上传本应被拦截。
- 现已修复：重复文件名判断改成大小写不敏感比较，确保同一业务日内仅大小写不同的同名 txt 也会被识别为重复上传。

### 58. 上传文件名解析之前错误区分 `.txt` 和 `.TXT`

- 文件名解析规则之前只接受小写 `.txt`，扩展名写成大写 `.TXT` 时会直接被判成“文件名格式不正确”。
- 这在 Windows 场景里很常见，也会让上一条“大小写不同的同名文件应视为重复”规则根本进不到重复判断阶段。
- 现已修复：文件名解析对 `.txt` 扩展名改为大小写不敏感，合法命名的 `.TXT` 文件现在会正常进入上传流程。

### 59. 后台上传页文件选择和拖拽过滤之前没有同步支持大写 `.TXT`

- 后端已经允许合法命名的 `.TXT` 文件上传，但后台上传页之前的文件选择框 `accept` 仍只写了 `.txt`，拖拽过滤也只接受小写 `.txt`。
- 这会导致 Windows 常见的 `.TXT` 文件虽然服务端能处理，前端却可能选不进队列或被拖拽过滤掉，形成前后端规则不一致。
- 现已修复：后台上传页文件选择框已改为同时接受 `.txt,.TXT`，拖拽过滤也统一按文件名小写后判断 `.txt`，和服务端解析规则保持一致。

### 60. Postgres 下 A 模式单张完成链路之前会因为变量名写错直接异常

- `complete_ticket()` 的 Postgres 分支之前把数据库更新结果保存到 `updated_rows`，但后续判断和返回却误写成了 `rows`。
- 这会导致 A 模式在真实 Postgres 环境下标记单张完成时直接抛 `NameError`，属于核心分票链路里的确定性故障。
- 现已修复：统一改为使用 `updated_rows` 控制文件计数更新和最终返回值，并补了回归测试覆盖这条单张完成路径。

### 61. SQLite 下文件完成计数之前会被 `assigned_count` 漂移持续带偏

- SQLite 路径里 `complete_ticket()`、`complete_tickets_batch()` 和 `finalize_tickets_batch()` 之前都把“文件 `assigned_count` 递减”和“`completed_count` 递增”绑在同一个条件 `assigned_count > 0` 上。
- 这意味着只要文件计数曾因为历史脏数据或边界故障出现过一次 `assigned_count=0` 漂移，后续真实完成的票也不会再增加 `completed_count`，文件状态和后台统计会被持续带偏。
- 现已修复：SQLite 路径改为自修复式更新，`assigned_count` 统一按 `max(current-1, 0)` 收敛到非负数，而 `completed_count` 只要票真实完成就继续累加，并补了 A/B 两条回归测试。

### 62. A 模式 Postgres 数据库兜底拿到禁止彩种时之前会把 Redis 队列重复塞脏

- A 模式 Postgres 分配逻辑支持两层候选来源：先从 Redis `pool:pending` 弹票，再用数据库 `FOR UPDATE SKIP LOCKED` 做兜底。
- 之前如果 Redis 没弹到票、但数据库兜底拿到的刚好是当前用户禁止接收的彩种，代码仍会无条件 `rpush` 回 Redis。
- 这张票本来就还在 Redis 队列里，兜底分支再 `rpush` 一次只会制造重复 `ticket_id`，长期会把 Redis 票池越堆越脏。
- 现已修复：只有真正从 Redis 弹出的禁止彩种票才会放回队列，数据库兜底候选不再重复回推，并补了回归测试。

### 63. B 模式票池状态接口之前显示的是原始待处理数，不是实际可下载数

- B 模式真实下载会全局预留 20 张票给 A 模式/上传缓冲，但 `/api/mode-b/pool-status` 之前直接透传原始 `pending` 状态统计。
- 这会导致页面显示“票池还有票、按钮可点”，但用户一点击下载又因为预留规则拿不到票，属于 B 模式主流程里的确定性前后端口径不一致。
- 现已修复：B 模式状态接口现在会按同样的 20 张预留规则裁剪 `total_pending` 和 `by_type`，页面显示和实际可下载范围保持一致，并补了回归测试。

### 64. 全局票池关闭之前不会真正拦住 A/B 分票入口

- 管理后台关闭 `pool_enabled` 之后，状态接口会显示票池为空，但 A 模式 `get_next_ticket()` 和 B 模式 `download_batch()` / `preview_batch()` 之前并没有同步检查这个全局开关。
- 这会出现“后台显示票池已关闭，但用户依然能继续接单/预览/下载”的确定性分票绕过，直接影响最核心的停池控制。
- 现已修复：A 模式在完成当前票后、领取下一张前会检查 `pool_enabled`；B 模式预查询和批量下载也都会在服务层拦截票池关闭状态，并补了回归测试。

### 65. B 模式预查询之前对已暂停接单用户仍会显示真实可用票数

- B 模式 `/api/mode-b/preview` 之前只校验登录态和模式，不看 `can_receive`。
- 这会导致管理员暂停某个用户接单后，该用户虽然实际无法下载，但预查询仍显示真实可用数，前端会误以为还能继续接单。
- 现已修复：B 模式预查询现在和票池状态接口保持一致，用户被暂停接单时统一返回 `available=0`、`sufficient=false`，并补了回归测试。

### 66. B 模式票池概况和预查询之前不会排除用户禁接彩种

- B 模式真实下载会按当前用户的 `blocked_lottery_types` 排除禁止接收的彩种，但 `/api/mode-b/pool-status` 和 `/api/mode-b/preview` 之前仍按全量票池统计。
- 这会导致用户明明被禁止接收某些彩种，页面却仍显示这些彩种和对应可下载张数，结果就是“看起来能拿、实际下载拿不到”。
- 现已修复：B 模式票池概况和预查询现在都会按当前用户的禁接彩种过滤后再做 20 张预留裁剪，页面展示和真实下载口径一致，并补了回归测试。

### 67. 用户首页统计和通用票池状态之前也会把禁接彩种算进去

- 除了 B 模式专用接口外，用户首页 `daily-stats` 里的 `pool_total_pending` 和通用 `/api/pool/status` 之前也都直接按全量票池统计。
- 这会让被配置了禁接彩种的用户在首页继续看到“剩余待处理很多”，但实际 A/B 分票时又拿不到这些彩种，仍属于核心票池口径不一致。
- 现已修复：用户首页统计和通用票池状态也都改为按当前用户的禁接彩种过滤后再统计，并补了回归测试。

### 68. B 模式用户首页顶部“票池剩余”之前仍显示未预留前的总待处理数

- 用户首页顶部统计来自 `/api/user/daily-stats`。之前即使 B 模式专用票池概况和预查询都已经按“预留 20 张”裁剪，这里的 `pool_total_pending` 仍直接取全量票池总数。
- 这会导致同一页面里出现自相矛盾的数字：顶部统计显示还能拿很多，但 B 模式实际下载和票池概况却只能拿预留后的少量票。
- 现已修复：`daily-stats` 对 B 模式用户改为按真实 `get_pool_total_pending()` 口径返回，即先排除禁接彩种，再减去预留 20 张；A 模式仍保持显示完整可接池，并补了回归测试。

### 69. 票池重新开启之前没有实时事件，用户页会一直停留在旧关闭状态

- 管理后台设置之前只会在关闭票池时推送 `pool_disabled`，重新开启时没有对应事件。
- 这会导致用户页即使后台已经重新开池，也要等到下一次轮询或手动刷新才能恢复显示和交互，属于核心停池控制的实时状态不完整。
- 现已修复：后台在 `pool_enabled` 切回 `true` 时会推送 `pool_enabled` 事件；Socket 客户端会分发这个事件，用户页收到后会和 `pool_disabled` 一样立即刷新票池状态与统计，并补了回归测试。

### 70. 通用 `/api/pool/status` 对 B 模式用户之前仍返回未预留前的总票池

- 虽然 B 模式专用 `/api/mode-b/pool-status` 已经按“预留 20 张”裁剪，但通用 `/api/pool/status` 之前仍直接返回全量待处理票数。
- 这会导致同一个 B 模式用户从不同接口看到两套数字，尤其首页统计和旧客户端若使用通用票池接口时会继续被误导。
- 现已修复：通用 `/api/pool/status` 现在也会在 B 模式用户场景下按相同的 20 张预留规则裁剪 `total_pending` 和 `by_type`，并补了回归测试。

### 71. WebSocket `request_pool_status` 之前仍会把全量票池推给 B 模式客户端

- 除了 HTTP 接口外，Socket 事件 `request_pool_status` 之前也直接调用全量 `get_pool_status()`，没有套用禁接彩种过滤和 B 模式 20 张预留规则。
- 这会让走 WebSocket 的客户端继续看到和 HTTP 接口不一致的票池数字，尤其是旧客户端或依赖 socket 主动刷新票池状态的场景。
- 现已修复：Socket 票池状态请求现在也会按当前用户的禁接彩种过滤，并在 B 模式用户场景下做同样的 20 张预留裁剪，和 HTTP 票池接口保持一致，并补了回归测试。

### 72. 用户首页之前没有监听 `pool_updated`，票池变化后会继续显示旧数量

- `static/js/socket_client.js` 早就会把 `pool_updated` 分发成自定义事件，但用户首页之前只监听公告、停池、开池和撤回，没有监听票池更新事件。
- 这会导致管理员上传新文件、其他人领票、系统批量过期等触发 `pool_updated` 后，用户首页顶部“待处理数量”和 B 模式票池概况仍继续显示旧值，直到 15 秒轮询或手动刷新。
- 现已修复：用户首页现在也会监听 `pool_updated`，收到后立即刷新当前用户自己的 `loadPoolStatus()` 和 `loadStats()`，避免直接吃全局广播值造成口径错乱，同时让票池变化更实时，并补了模板回归测试。

### 73. 用户首页两个加载失败分支之前写反了，网络抖动后会留下错模块的旧数据

- `loadStats()` 的失败分支之前误把 `bPendingBatches` 清空了，但不会重置顶部统计；`loadProcessingBatches()` 的失败分支则反过来误把 `stats` 清空。
- 这会导致接口偶发失败时，页面出现“统计还是旧值、处理中列表突然没了”或者“批次列表还在、顶部统计被清零”的错位状态，尤其在票池高频变化和网络抖动时很容易误导用户。
- 现已修复：`loadStats()` 失败时只重置统计，`loadProcessingBatches()` 失败时只清空处理中批次，并补了模板回归测试。

### 74. A 模式“获取下一张”HTTP 失败时之前把错误写进了密码弹窗状态

- 用户首页 `nextTicket()` 在 `/api/mode-a/next` 返回非 200 时，之前误把错误写到 `pwdError`，这本来是修改密码弹窗专用状态。
- 这会导致 A 模式真失败时页面不弹出票提示，用户看起来像“点了没反应”，同时密码弹窗状态还会被莫名污染。
- 现已修复：A 模式“获取下一张”在 HTTP 失败时现在直接 `showToast` 提示实际错误，不再写入密码弹窗状态，并补了模板回归测试。

### 75. Postgres 下完成票据时文件 `assigned_count` 之前可能被减成负数

- 之前 `services/ticket_pool.py` 在 Postgres 路径里，无论是 A 模式单张完成、B 模式批量完成，还是 B 模式“部分完成其余过期”，更新 `uploaded_files` 时都直接做 `assigned_count - 1` 或 `assigned_count - sub.cnt`。
- 一旦文件计数此前已经漂移到 0，这几条路径就会把 `assigned_count` 继续减成负数，进一步污染文件派生状态、后台文件列表和后续计数修复。
- 现已修复：Postgres 下相关完成/过期链路现在统一把 `assigned_count` 下限钳到 0，A 模式单张完成用 `CASE ... ELSE 0`，B 模式批量完成和部分过期用 `GREATEST(..., 0)`，并补了回归测试覆盖这三条路径。

### 76. 后台文件列表之前会把“已过截止但仍有处理中票”的文件误显示成已过期

- `UploadedFile.to_dict()` 返回的 `status` 已经是后端派生状态，会综合 `pending_count`、`assigned_count`、`completed_count` 和撤回状态判断。
- 但后台文件列表模板之前没有信任这个后端状态，而是自己用“`completed_count === total_tickets` 或 `deadline_time < now`”重新算一遍。
- 这会导致文件里明明还有 `assigned` 票在处理，只要过了截止时间，前端就直接显示“已过期”，误导管理员判断当前分票/出票进度。
- 现已修复：后台文件列表现在直接按后端返回的派生状态渲染 `revoked / exhausted / expired / active`，不再前端二次猜测，并补了模板回归测试。

### 77. 后台上传页整体上传失败时之前会把队列静默清掉，像是“上传成功但没入库”

- 管理后台上传页 `doUpload()` 之前没有检查 `/admin/files/upload` 的 HTTP 状态码，只在网络异常时才进 `catch`。
- 这会导致接口整体返回 400/401/403 时，前端仍继续往下走，既不弹出实际错误，也可能把队列误当成已处理完成，造成“看起来上传过了、实际上没进库”的错觉。
- 现已修复：上传页现在会先检查 `!res.ok || data.success === false`，整体失败时直接抛出错误，保留失败文件在队列里并回填错误消息，同时给出 toast 提示，并补了模板回归测试。

### 78. 后台文件详情和撤回之前也没统一处理 HTTP 失败，前端会走半成功分支

- 后台上传管理页的 `viewDetail()` 和 `revokeFile()` 之前只在网络异常时走 `catch`，对接口返回的 400/404/500 没有统一校验。
- 这会导致后端已经明确返回错误时，前端还继续按成功分支往下跑，文件详情可能弹空表，撤回失败也可能只给出模糊提示。
- 现已修复：文件详情和撤回现在也统一检查 `!res.ok || data.success === false`，优先展示后端返回的实际错误信息，并补了模板回归测试。

### 79. 定时过期任务之前不会推送 `pool_updated`，票池变化后前端会继续显示旧数量

- `expire_overdue_tickets()` 之前只会通知管理员“有多少票过期了”，但不会同步广播新的票池状态。
- 这会导致定时任务把一批 `pending/assigned` 票改成 `expired` 后，用户页和管理员页仍继续显示旧的票池数量，直到 15 秒轮询或手动刷新。
- 现已修复：过期任务在成功提交后，如果本轮确实有票过期，会同步推送一次最新 `pool_updated`，并补了回归测试。

### 80. 用户首页之前不分 A/B 模式就去调 B 模式接口，A 模式用户会持续打 403

- 用户首页 `mounted()` 之前无论当前账号是 A 模式还是 B 模式，都会直接调用 `loadPoolStatus()` 和 `loadProcessingBatches()`，而这两个接口都是 `/api/mode-b/*`。
- 这会导致 A 模式用户在页面初次加载、实时事件触发和 15 秒轮询时持续打到 B 模式受限接口，产生无意义的 403 和错误分支。
- 现已修复：用户首页现在只会在 `client_mode == mode_b` 时调用 B 模式票池和处理中接口；A 模式用户仍会正常刷新自己的顶部统计，不再碰 B 模式专用接口，并补了模板回归测试。

### 81. 票池全局关闭时，用户首页顶部“待处理数量”之前仍会显示真实 pending 数

- `/api/user/daily-stats` 之前计算 `pool_total_pending` 只看 `can_receive` 和当前模式，没有把全局 `settings.pool_enabled` 纳入判断。
- 这会导致管理员已经关闭票池后，用户首页顶部仍然显示还有多少票待处理，看起来像“还有票可接”，但实际所有分票入口都已经被拦住。
- 现已修复：用户首页统计现在只有在“用户可接单且全局票池开启”时才显示待处理数量；票池关闭时统一返回 `0`，并补了回归测试。

### 82. B 模式 `preview` 之前按总票数算，可能显示“够下”但实际一批拿不到那么多

- B 模式 `download_batch()` 的真实规则是：一次只会分配同一彩种、同一截止时间的一批票，不会混合多个批次凑数。
- 但 `preview_batch()` 之前只看“总 pending 减预留 20 张”后的数量，所以会出现总票数够、预览显示 `sufficient=true`，实际下载却只能拿到更少的情况。
- 现已修复：B 模式预览现在改为按和 `assign_tickets_batch()` 相同的单批选择规则计算“当前最多能拿多少张”，保证预览和真实下载口径一致，并补了回归测试。

### 83. B 模式首页两个状态接口之前对 HTTP 错误不清状态，会把旧数据留在页面上

- 用户首页的 `loadProcessingBatches()` 和 `loadPoolStatus()` 之前只在网络异常时走 `catch`，如果接口返回的是 JSON 形式的 401/403/400，则不会抛错，也不会清空旧状态。
- 这会导致 B 模式用户在权限被撤销、会话失效或服务器明确返回错误后，页面仍继续显示上一次的处理中批次和票池数量，形成明显假象。
- 现已修复：这两个方法现在都会统一检查 `!res.ok || data.success === false`，失败时主动重置对应状态，并补了模板回归测试。

### 84. 后台批量上传接口之前即使全失败也返回 200，外部上传程序很难判断整批失败

- `/admin/files/upload` 之前无论每个文件实际是否都失败，顶层都会返回 `success: true` 和 HTTP 200。
- 这会导致后台前端、外部自动上传脚本和后续联动很难准确区分“本次至少有一份成功”还是“整批全部失败”，尤其是同名文件重复上传这种场景。
- 现已修复：后台上传接口现在会在“本次至少有一份成功”时返回 200；如果本次所有文件都失败，则返回 400 和 `success:false`，同时保留逐文件 `results` 明细，并补了回归测试。

### 85. TXT 上传文件如果既不是 UTF-8 也不是 GBK，之前会直接打成 500

- `process_uploaded_file()` 之前只做了“先按 UTF-8 读，失败后改按 GBK 读”。
- 如果文件内容两种编码都无法解码，第二次读取抛出的 `UnicodeDecodeError` 会直接冒泡，后台上传接口会变成 500，属于上传主链路里的确定性异常。
- 现已修复：两种编码都失败时会回滚本次上传、删除临时文件，并稳定返回“文件编码无法识别，请使用 UTF-8 或 GBK”，同时补了回归测试。

### 86. Postgres 分票更新文件计数之前还不是自修复式，历史漂移后会继续带脏

- A 模式单张分票和 B 模式批量分票在 Postgres 路径下，之前更新 `uploaded_files` 时仍然直接做 `pending_count - 1` / `pending_count - sub.cnt`。
- 这会导致一旦文件计数曾因历史脏数据或边界故障发生漂移，后续真实分票会继续把 `pending_count` 减成负数，或者因为旧条件卡住导致 `assigned_count` 不再跟上真实分配。
- 现已修复：Postgres 分票链路现在统一改成 `GREATEST(pending_count - ..., 0)` 的自修复式更新，既不会减成负数，也不会因为旧漂移阻断新分票的文件计数回写，并补了 A/B 两条回归测试。

### 87. 后台批量上传里只要单个文件抛异常，之前整批就会直接 500

- `/admin/files/upload` 之前逐个调用 `process_uploaded_file()`，但没有兜住单文件级别的未预期异常。
- 这会导致同一批里只要有一份文件触发未覆盖异常，整个接口直接 500，后面的文件也不会继续处理，管理员很难判断哪些已经成功、哪些失败。
- 现已修复：后台批量上传现在会把单文件异常收敛成对应 `results` 里的失败项，继续处理同批剩余文件；只在“整批全失败”时返回 400，否则仍返回 200，并补了回归测试。

### 88. TXT 内部某一行金额解析失败时，之前会按 0 元照样入池

- `process_uploaded_file()` 之前对每一行用 `calculate_ticket_amount(line) or Decimal('0')`，这意味着只要某一行内容格式不合法，金额解析失败就会被静默当成 `0` 元。
- 这会把脏票行直接混进票池，污染金额统计、文件总金额和后续分票链路，属于上传入池主链路里的确定性错误。
- 现已修复：只要文件中任意一行内容格式无效，整份文件都会回滚失败，并明确返回具体行号；不会再出现“部分脏行按 0 元混入系统”的情况，同时补了回归测试。

### 89. 文件撤回已经落库后，如果实时通知失败，之前整个撤回接口会被打成 500

- `revoke_file()` 之前在完成数据库提交和 Redis 清理之后，直接调用 `notify_all('file_revoked', ...)`，没有再做异常兜底。
- 这会导致一种很危险的假象：文件和票据其实已经撤回成功，但只要 WebSocket 通知层瞬时异常，管理员接口就会直接 500，看起来像“撤回失败”。
- 现已修复：撤回通知现在改成 best-effort，通知失败只记录 warning，不再影响已经成功提交的撤回结果，并补了回归测试。
