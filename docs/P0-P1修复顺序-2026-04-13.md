> 原则：**先修会阻断系统稳定运行的问题，再修长期优化项**

## 0. 与主计划的关系

本文件不是接手入口，也不是 `P0-1` 之前的执行入口。  
只有在主计划确认已经完成 `P0-1` 基线验证，并且 `docs/真实bug与风险清单-2026-04-13.md` 已经形成真实问题条目后，才允许用本文件决定修复顺序。

后续 AI 使用本文件前，必须先完成：

1. 按 `docs/小项目长期稳定性计划-主计划.md` 的 `0.1 AI 最小接手读取协议` 确认当前进度。
2. 按主计划 `0.2 阶段信息足够性门禁` 确认信息足够。
3. 读取主计划 `6.3 P0-2 核心业务链路修 bug`，确认修复阶段目标、边界、固定动作和完成标准。
4. 读取真实 bug / 风险清单中与本轮修复相关的条目。

本文件只负责给真实问题排序。  
如果本文件与主计划冲突，以主计划为准。

本文件中的默认顺序只能作为候选顺序，不能替代真实验证证据。  
没有复现证据和清单条目的问题，不允许进入正式修复。

## 一、排序原则

任何问题的修复顺序，必须满足下面的优先级逻辑：

1. 先修会影响核心业务正确性的
2. 先修会造成数据污染的
3. 先修高频问题
4. 先修可复现问题
5. 先修影响大范围用户的问题
6. 最后才修边缘体验问题

不允许因为某个问题“看起来更优雅”就提前。

## 二、第一批默认修复顺序

下面是第一批修复的默认排序参考。  
它只用于 `P0-1` 基线验证结束、真实 bug 清单形成后的排序，不代表可以在基线验证前提前开始修复。

## 第一优先级：认证与会话

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- 登录失败率高
- 会话容易失效
- 心跳刷新异常
- 设备数限制异常

### 原因

登录和会话是一切业务的入口。入口不稳，后面所有链路验证都会被污染。

### 涉及文件

- `routes/auth.py`
- `services/session_service.py`
- `app.py`
- `models/user.py`

## 第二优先级：上传与拆票

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- 上传后数据不一致
- 重复上传防护失效
- 文件解析金额或张数错误
- 上传失败后残留脏文件 / 脏记录

### 原因

上传与拆票是业务源头，源头错了，后续全错。

### 涉及文件

- `services/file_parser.py`
- `models/file.py`
- `models/ticket.py`
- `routes/admin.py`

## 第三优先级：分票与状态流转

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- 重复分票
- 串设备
- assigned 残留
- completed / pending / expired 流转错误

### 原因

分票核心逻辑直接决定业务正确性，是系统最关键的内部机制之一。

### 涉及文件

- `services/ticket_pool.py`
- `services/mode_a_service.py`
- `services/mode_b_service.py`
- `routes/mode_a.py`
- `routes/mode_b.py`

## 第四优先级：赛果与中奖图

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- 赛果上传后不触发计算
- 重算行为错误
- 中奖图片错绑
- 图片记录与票据状态不同步
- 替换图片时误删旧图

### 原因

这条链路一旦错，会直接伤害业务结果可信度。

### 涉及文件

- `routes/winning.py`
- `services/winning_calc_service.py`
- `services/oss_service.py`
- `routes/admin.py`

## 第五优先级：用户统计与导出

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- 日统计明显错误
- 导出结果不可信
- 导出大面积失败

### 原因

这部分虽然不是最底层核心，但它直接影响管理和对账。

### 涉及文件

- `routes/user.py`
- `routes/admin.py`
- `models/ticket.py`

## 第六优先级：调度器与定时任务

### 适用条件

如果验证发现以下任意问题成立，优先修这里：

- scheduler 没启动
- 关键任务没注册
- 关键任务执行失败但不易发现

### 原因

调度器对长期运行重要，但前提是核心主链路已经基本可用。

### 涉及文件

- `tasks/scheduler.py`
- `tasks/expire_tickets.py`
- `tasks/clean_sessions.py`
- `tasks/daily_reset.py`
- `tasks/archive.py`
- `app.py`

## 三、明确后置项

以下内容即使存在问题，也默认后置，除非验证证明它们已经升级为 P0 / P1：

1. 文案优化
2. 低频 UI 小问题
3. 非核心统计展示细节
4. 复杂后台治理功能
5. 复杂观测体系
6. 多余的长期制度设计

## 四、每轮修复的固定动作

后续每修一项，必须按这个顺序做：

1. 先从真实 bug 清单中选一项
2. 写明这项为什么是当前最高优先级
3. 再修改代码
4. 修改后做最小验证
5. 把结果写入执行记录
6. 在主计划里打勾或更新状态

不允许跳过记录直接做下一项。

## 五、主计划勾选规则

### 可以打勾的前提

只有当某一项满足以下条件时，才允许在主计划中打勾：

1. 已完成实际修改
2. 有验证结果
3. 风险说明已写入执行记录

### 不允许打勾的情况

以下情况不允许打勾：

1. 只是分析过
2. 只是猜测修复了
3. 只是改了文档
4. 只是修了部分现象

## 六、修复节奏建议

推荐节奏如下：

1. 每一轮只处理 1 到 2 个明确问题
2. 每轮结束必须留记录
3. 每轮结束都要写下一步
4. 不要一轮同时改很多主链路

原因：

1. 便于定位回归
2. 便于控制风险
3. 便于后续 AI 接手
